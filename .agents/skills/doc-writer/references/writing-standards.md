# Torus Documentation Writing Standards

## Priorities
1. Clarity
2. Correctness
3. Conciseness
4. Completeness for intended audience

## Style Rules
- Use plain language and active voice.
- Prefer short paragraphs and single-level lists.
- Define domain terms on first use.
- Keep terminology consistent across docs.
- Avoid speculative statements unless explicitly labeled as assumptions.

## Structure Rules
- Start with scope and intended audience.
- Include prerequisites and constraints before procedures.
- Document success criteria and expected outcomes.
- Include failure modes and recovery paths for non-trivial workflows.

## Torus Terminology Baseline
- `Project`: authoring workspace for course content.
- `Publication`: immutable published snapshot.
- `Section`: delivery instance for learners/instructors.
- `Resource/Revision`: versioned content pattern.
- `Enrollment`: user membership in a section.
- `Attempt`: learner interaction record for activities.

## Accuracy Checklist
- Verify technical claims against code and existing docs.
- Include exact module/file references for technical details.
- Separate current behavior from planned behavior.
- Note role permissions where applicable.

## Readability Checklist
- Use descriptive headings.
- Keep sections independently understandable.
- Prefer examples that mirror real Torus flows.
