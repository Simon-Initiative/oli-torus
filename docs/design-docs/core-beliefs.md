# Core Beliefs

- Simplicity is the default design goal. Prefer the simplest model, workflow, and implementation that preserves correctness and future growth.
- Authoring and delivery are different responsibilities and should remain clearly separated in the product model.
- Published learner-facing content must be stable. Author progress should not destabilize live course delivery.
- Versioned content is a feature, not an implementation detail. Changes should be explicit, traceable, and reversible through revisions and publications.
- Users should not need to understand internal platform complexity to accomplish common work.
- Torus should feel approachable to users who are not experts in learning engineering, while still supporting advanced instructional practice.
- The platform should integrate well with LMS ecosystems instead of trying to replace them wholesale.
- Extensibility matters, especially for learning activities and future instructional patterns, but it should not come at the cost of clarity.
- Operational scalability and runtime reliability are product requirements, not afterthoughts.
- Accessibility is a core quality bar. Interfaces should be usable, perceivable, and operable by default.
- Responsive design is a baseline expectation. Core workflows should work across supported devices and viewport sizes.
- Product behavior should favor predictability over surprise, especially around publication, grading, attempts, and learner progress.
- Shared concepts should stay consistent across the system. Projects, resources, revisions, publications, sections, and attempts should mean the same thing everywhere.
- Auditability and observability matter because the platform supports real institutions, real courses, and real learners.
