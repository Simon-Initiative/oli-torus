# Welcome to Torus Course

## Unit 1: Authoring Detailed Design

Last updated: 2026-03-26

This document expands Unit 1 from the outline in `overview.md`. It is intended to be the Phase 2 detail pass for the Authoring unit and to serve as the source material for later page authoring and MCP-based activity creation.

## Unit Goal

By the end of Unit 1, a learner should be able to:

- explain the difference between a project, a page, an activity, and a publication
- navigate the main authoring workspace areas
- organize a course into units, modules, and pages
- add formative activities to instructional pages
- use preview and review workflows before publishing
- explain how publishing affects delivery without confusing authoring with live sections

## Unit Vocabulary

Use these terms consistently throughout the unit:

- Project
- Curriculum
- Page
- Activity
- Resource
- Revision
- Publication
- Collaborator
- Preview
- Review
- Publish

## Module 1.1: Getting Started in Authoring

### Module Goal

Help a new author understand the authoring model, the main workspace surfaces, and the publication-aware content lifecycle.

### Learning Objectives

After this module, the learner should be able to:

- identify the core authoring surfaces in Torus
- describe how a project contains curriculum, pages, and activities
- explain the role of revisions and publications in authoring
- describe basic collaboration and locking expectations

### Page Plan

#### Page 1: What Torus Authoring Is

Summary:

- Introduce Torus as a course-authoring platform, not just a page editor.
- Explain that authors work inside projects and use those projects to build instructional structure and assessment content.
- Set up the distinction between authoring work and learner-facing delivery.

Key teaching points:

- authors build in projects
- projects contain structured learning materials
- activities are part of the authored learning experience
- delivery uses published output, not raw authoring state

Suggested formative activity:

- Type: `oli_multiple_choice`
- Prompt idea: Which statement best describes Torus authoring?
- Answer intent: it is a project-based environment for building and revising course materials before delivery

#### Page 2: Navigating the Authoring Workspace

Summary:

- Tour the major authoring entry points visible in the router and workspace code paths.
- Show learners how curriculum, pages, activities, review, preview, objectives, bibliography, and publish fit together.
- Focus on orientation rather than exhaustive feature depth.

Key teaching points:

- curriculum is where course structure is organized
- pages and activities can be browsed as distinct resource sets
- preview is for checking learner-facing experience
- review is for QA before publish
- publish is the transition from mutable authoring work to stable delivery content

Suggested formative activity:

- Type: `oli_check_all_that_apply`
- Prompt idea: Which workspace areas would an author commonly use before publishing a project revision?
- Answer intent: curriculum, pages, activities, preview, review, publish

- `[Screenshot placeholder: authoring navigation with curriculum, pages, activities, review, publish]`

#### Page 3: Understanding Resources, Revisions, and Publications

Summary:

- Explain the publication model in plain language.
- Show how a resource can have multiple revisions and how publications freeze a learner-facing version.
- Make this the conceptual hinge page for the whole course.

Key teaching points:

- resources are durable containers for course content
- revisions capture change over time
- publications freeze a stable version set
- sections point at publications, not ongoing draft edits

Suggested formative activity:

- Type: `oli_ordering`
- Prompt idea: Put these in workflow order: edit content, create revision changes, publish, teach from section
- Answer intent: edit -> revision state -> publish -> teach from section

Media:

- `[Diagram placeholder: resource to revision to publication to section flow]`

#### Page 4: Collaborating Safely

Summary:

- Introduce collaborator workflows and the need for editing discipline.
- Explain that shared authoring requires clear ownership and awareness of locking.
- Keep this operational and lightweight.

Key teaching points:

- projects can have collaborators
- editing workflows need coordination
- locks exist to reduce conflicting edits
- authors should preview and review before changing shared production materials

Suggested formative activity:

- Type: `oli_short_answer`
- Prompt idea: Why is it useful for collaborators to coordinate page ownership before editing?
- Expected answer intent: prevent conflicting edits, reduce confusion, and maintain content quality

## Module 1.2: Building Course Structure

### Module Goal

Teach learners how to organize a course into a coherent hierarchy of units, modules, and pages.

### Learning Objectives

After this module, the learner should be able to:

- describe the purpose of units, modules, and pages in course structure
- organize content into a logical curriculum sequence
- create pages that support learning flow
- place formative and scored work intentionally inside the structure

### Page Plan

#### Page 1: Designing a Course Hierarchy

Summary:

- Introduce course structure as a learner-navigation problem, not only an authoring convenience.
- Show how units, modules, and pages should reflect learning goals and progression.

Key teaching points:

- units group major themes
- modules group related lessons or workflows
- pages deliver instruction and interaction
- structure should make navigation predictable

Suggested formative activity:

- Type: `oli_ordering`
- Prompt idea: Arrange the course hierarchy from largest to smallest.

#### Page 2: Creating and Organizing Curriculum Containers

Summary:

- Focus on the curriculum surface and how authors use it to create structure.
- Explain container editing, placement, and organizational hygiene.

Key teaching points:

- curriculum is the structural backbone of the course
- container organization affects learner navigation and author maintainability
- consistent naming improves both authoring and delivery clarity

Suggested formative activity:

- Type: `oli_check_all_that_apply`
- Prompt idea: Which practices improve course structure clarity?

Media:

- `[Screenshot placeholder: curriculum editor showing unit/module nesting]`

#### Page 3: Creating Pages for Learning Flow

Summary:

- Show how pages function as the primary instructional unit.
- Explain page naming, scope, and sequencing.
- Encourage one clear teaching purpose per page.

Key teaching points:

- pages should have a clear instructional goal
- page titles should be descriptive and consistent
- page order should reflect how a learner progresses through a topic
- long workflows should be broken into manageable pages

Suggested formative activity:

- Type: `oli_short_answer`
- Prompt idea: What makes a page title useful in a course with many modules?

#### Page 4: Planning Assessments in the Structure

Summary:

- Explain where to place practice and where to place scored checks.
- Tie assessment placement to module closure and learner expectations.

Key teaching points:

- formative activities belong inside learning pages
- scored assessments should close a module
- placement should match the learner journey, not just author convenience

Suggested formative activity:

- Type: `oli_multiple_choice`
- Prompt idea: Where should a module-level scored assessment usually appear?

## Module 1.3: Authoring Content, Activities, and Publishing

### Module Goal

Teach learners how to create instructional pages, add formative practice, check their work, and publish responsibly.

### Learning Objectives

After this module, the learner should be able to:

- draft instructional pages with clear teaching intent
- choose appropriate formative activity types for a page
- describe how preview and review support content quality
- explain when and why to publish

### Page Plan

#### Page 1: Building Effective Instructional Pages

Summary:

- Show what makes a page instructional instead of merely informational.
- Encourage concise explanations, visible learner tasks, and a practical flow.

Key teaching points:

- pages should combine explanation with action
- content should be scoped to one job or concept cluster
- media should support instruction, not decorate it

Suggested formative activity:

- Type: `oli_check_all_that_apply`
- Prompt idea: Which design choices improve an instructional page?

Media:

- `[Screenshot placeholder: strong example of a compact instructional page]`

#### Page 2: Adding Formative Activities

Summary:

- Introduce the built-in activity set chosen for this course design.
- Show what each activity type is good for.

Key teaching points:

- use `oli_multiple_choice` for recognition and decision checks
- use `oli_check_all_that_apply` for multi-condition judgment
- use `oli_ordering` for workflows and sequences
- use `oli_short_answer` for explanation and recall in the learner's own words

Suggested formative activity:

- Type: `oli_multiple_choice`
- Prompt idea: Which activity type is best for checking whether a learner knows the steps of a publish workflow?
- Answer intent: ordering

Media:

- `[Screenshot placeholder: activity creation or activity bank flow]`

#### Page 3: Previewing and Reviewing a Project

Summary:

- Explain preview as a learner-facing check and review as a quality pass.
- Make clear that authors should inspect both content correctness and learner experience.

Key teaching points:

- preview helps catch presentation and flow issues
- review helps catch readiness and quality issues
- authors should inspect pages and activities before publish

Suggested formative activity:

- Type: `oli_short_answer`
- Prompt idea: What is one issue an author might catch in preview before publishing?

#### Page 4: Publishing and Updating Content

Summary:

- Explain publish as the handoff from authoring change to stable course version.
- Reinforce that publication protects active delivery from draft instability.

Key teaching points:

- publishing freezes a version set
- authors can continue improving the project after a publish
- downstream delivery contexts depend on publication state

Suggested formative activity:

- Type: `oli_ordering`
- Prompt idea: Order the pre-publish workflow from draft editing through publish.
